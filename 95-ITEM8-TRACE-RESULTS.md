## Item 8 Detailed Trace

### What the drawing shows
The catalog field for Item 8 contains two part numbers in one cell:
- **TYD15X3WPW6** — Thomas & Betts wide slot wiring duct
- **WITH COVER TYD2CW6** — the duct cover accessory

No "/" separator exists in the source.

### What happened (proven by evidence)

**Stage A — Model output:** The model misread "TYD15X3WPW6" and returned a compound PN containing "/" (exact compound unknown — raw output not stored). One segment was "MPWS". The model also returned `additionalPartNumbers: [{partNumber: "TYD2CWS", relationship: "accessory", context: "WITH COVER TYD 2CWS in catalog field"}]` — misreading TYD2CW6 as TYD2CWS (6→S).

**Stage B — Slash-split (app.jsx:11643):** Code split the compound PN at "/" into segments. Segment 0 = unknown main part. Segment 1 = "MPWS" with "(sub-part from above)" appended to description.

**Stage C — Positional dedup (app.jsx:10629):** Both segments shared identical y_top=0.916, y_bottom=0.935, sourcePageIdx=0 (inherited via `{...item}` spread). Positional dedup merged them as "same row." MPWS won the score contest because its description was ~22 chars longer (the "(sub-part from above)" suffix). **The main part number was destroyed.**

**Stage D — Companion split (app.jsx:10826):** Found TYD2CWS in MPWS's `additionalPartNumbers` array. Created companion row with `autoAddedCompanion: true`.

**Stage E — No downstream transforms:** isCrossed, isCorrection, correctedByLibrary all undefined. priceSource: "ai". No ARC Cross, no corrections library, no BC lookup touched either row.

### Code-path bug: slash-split × positional-dedup
This is a **deterministic** destructive interaction:
1. Slash-split creates N rows with identical Y coordinates
2. Positional dedup merges rows at same Y
3. The sub-part always wins (longer description)
4. The main part is always destroyed

This will fire on EVERY compound PN the model returns with a "/" — regardless of whether the source drawing actually contains a slash.

## Error Classification

### Class 1: Digit substitution (Items 3, 5, 7)
Single-digit swaps between confusable glyphs. Occurs on pdf-native path because the model reads visually. Not specific to scanned/degraded documents.
- 3→0, 8→6, 8→9

### Class 2: Structural misread (Items 8, 12, 13)
Wholesale part number restructuring. The model doesn't just swap digits — it produces a plausible-looking but entirely wrong catalog number.
- TYD15X3WPW6 → MPWS (+ pipeline bug)
- LNM25BPK100 → LNMQ3RP-100 (phantom Q insertion, B→R, K→hyphen)
- LNM40BPK100 → LNMQ8RP-100 (same pattern)

Class 2 is more dangerous than Class 1 — BC validation can catch Class 1 errors IF the wrong digit produces an invalid PN, but Class 2 errors may land on a different valid catalog item entirely.

## Recommendations for Freddy

1. **The slash-split × dedup bug is a code fix** — Coach should verify the mechanism, then Marc implements. Possible fixes: skip positional dedup on slash-split siblings, or don't split compound PNs at all (let the AI handle them as single items).

2. **The 46% error rate on this drawing is concerning** but may not be representative — this is a single 13-item BOM from one drawing format. Before concluding the extraction pipeline is broadly broken, test against other recent projects to establish a baseline error rate.

3. **Raw model output should be stored** — the trace was severely hampered by not having the raw AI response. Storing `data.raw` (or a parsed summary with per-item original values) in `extractionReport` would make future diagnostics trivial.

4. **The "vision on vector PDF" finding** means the existing confidence-downgrade logic (confusable-glyph detector at app.jsx:11617) is correctly scoped — it should remain active even on pdf-native extractions.
