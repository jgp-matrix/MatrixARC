# Multi-Role Startup & Onboarding System — Design

**Author:** Sam Wize (Coach)
**Date:** 2026-06-02
**Status:** DRAFT — awaiting Jon's review

---

## Problem Statement

When Jon starts a new work session, each Claude role needs context. Coach and Marc can read repo files, but Freddy (Claude.ai browser) has no repo access. Today Jon manually re-explains context to each new session. This wastes time and introduces drift when Jon misses details.

Jon needs a system where:
1. Marc's CCD session starts and asks which roles are active
2. Based on the answer, Marc generates onboarding pastes for the other roles
3. Coach self-orients from repo files (no paste needed)
4. Freddy receives a paste from Jon (who copies it from Marc's output)

---

## 1. File Structure

### Existing files (no changes needed)

| File | Purpose | Maintained by |
|------|---------|---------------|
| `CLAUDE.md` | Shared startup context for all Claude instances. Role definitions, conventions, procedures. | Marc (CCD) |
| `COACH.md` | Coach's session log — findings, verifications, sign-offs. NOT an onboarding doc. | Coach |
| `FREDDY.md` | Static role identity and knowledge base for Freddy. Conventions, workflow, project background. | Coach |
| `TODO.md` | Running findings list with OPEN/RESOLVED/STALE status. | Marc (during Close Out) |

### New files

| File | Purpose | Maintained by |
|------|---------|---------------|
| `SESSION-STATE.md` | Current-state snapshot: version, recent deploys, active work, blockers, audit status. Generated at session end, refreshed at startup. | Generated by Marc or Coach |

### Files NOT created (and why)

- **No MARC.md** — Marc reads CLAUDE.md on startup automatically. His role identity is already defined there (line 188). A separate file would duplicate CLAUDE.md and drift.
- **No COACH-ONBOARD.md** — Coach has repo access and self-orients from CLAUDE.md + SESSION-STATE.md + git log. A paste-based onboarding doc would be redundant.

## 2. Static Files vs On-Demand State

### Static (rarely changes, lives in files)

- Role identity — who Freddy/Coach/Marc are, what they do
- Communication conventions — notification prefixes, paste format, tool restrictions
- Project background — what ARC is, the stack, BC integration
- Workflow pipeline — Brief → Supplement → Review → Plan → Implement

These live in FREDDY.md (for Freddy) and CLAUDE.md (for Coach/Marc). Updated only when conventions change, maybe once per milestone.

### Dynamic (changes every session, generated fresh)

- Current APP_VERSION
- Last 10 commits (what shipped recently)
- Active CRITICAL/HIGH findings and their status
- In-flight work (uncommitted changes, open branches)
- Blockers or waiting-on items
- TODO.md OPEN count

This is SESSION-STATE.md. It's generated, not hand-written.

### Token budget for Freddy paste

The Freddy paste combines FREDDY.md + SESSION-STATE.md. Target sizes:

| Component | Target | Content |
|-----------|--------|---------|
| FREDDY.md (role + conventions) | ~2,000 tokens | Static — trimmed to essentials |
| SESSION-STATE.md (current state) | ~800 tokens | Generated — compact bullet format |
| **Total paste** | **~2,800 tokens** | Fits easily in Claude.ai context |

This is small enough to paste quickly and leaves Freddy's full context window for conversation.

## 3. SESSION-STATE.md — Format and Generation

### Format

```markdown
# Session State — [date] [time] MDT

## Version
v1.20.74 (deployed [date])

## Recent Commits (last 10)
- cafbd49e docs: add FREDDY.md for new-session continuity
- 15d9b894 Release v1.20.74
- 9e0e3fe2 Fix F-1c.1: resolve vendor/customer names from BC on restore remap
- ...

## Active Work
- [CRITICAL] F-2d.1 — 46 BC fetch calls bypass semaphore (not started)
- [CRITICAL] F-1g.1 — "AI missed" message misleading for dedup gaps (not started)
- [CRITICAL] F-2d.2 — PDF attachment not atomic (not started)
- [CRITICAL] F-2d.3 — Firestore write after BC sync crash gap (not started)
- [DONE] F-1c.1 — Vendor name nulled on restore remap (shipped v1.20.74)
- [DONE] Marc Independent #1 — Stale init.panels to BC sync (shipped v1.20.71)

## Blockers
- BOM prompt duplicate-merge instruction (app.jsx:11265, bomPrompt.js:215) — root cause of 592273 dedup failure. Prompt amendment needed. No fix deployed.

## Working Tree
- Clean / N uncommitted files
- Branch: master

## Open TODOs
N OPEN findings in TODO.md (of M total)
```

### Generation procedure

Marc or Coach runs this at session end (or on demand at startup):

1. Read `APP_VERSION` from `public/index.html`
2. Run `git log --oneline -10`
3. Read ARC-AUDIT-FINDINGS.md executive summary (first 40 lines) + cross-reference against recent commits to determine which findings are fixed
4. Run `git status` for working tree state
5. Count OPEN items in TODO.md
6. Check OVERNIGHT-LOG.md for any unresolved items
7. Write SESSION-STATE.md

This takes ~15 seconds. No manual writing.

### When to generate

| Trigger | Who generates | Why |
|---------|---------------|-----|
| Session Close Out | Marc (automatic as part of Close Out procedure) | Captures state for next session |
| Session startup (if stale) | Marc (before generating pastes) | Refresh if >24h old |
| On demand ("refresh state") | Marc or Coach | Jon asks for it |

**Staleness check:** If SESSION-STATE.md's date header is older than the latest commit, regenerate before using it for pastes.

## 4. Startup Flow

### Trigger

Jon's first message to CCD includes "startup" (or Marc detects it's a fresh session with no prior context). This is a behavioral instruction in CLAUDE.md, not a code hook.

### Flow diagram

```
Jon opens CCD
       │
       ▼
Marc reads CLAUDE.md (automatic)
       │
       ▼
Marc runs verify-state.sh (existing procedure)
       │
       ▼
Marc reads SESSION-STATE.md
  ├─ exists and fresh? → use it
  └─ missing or stale? → regenerate from repo
       │
       ▼
Marc asks Jon: "Which roles this session?"
  ┌────┼────────┬──────────┐
  ▼    ▼        ▼          ▼
 Full  Code    Quick     Solo
 Team  Team     Fix
  │    │        │          │
  ▼    ▼        ▼          ▼
 F+C+M C+M     F+M        M
       │
       ▼
For each active role:
  ├─ Freddy? → Generate paste, display in code block
  ├─ Coach?  → Note: "Coach will self-orient from files"
  └─ Marc?   → Already active (this session)
       │
       ▼
Marc displays:
  1. Current state summary (from SESSION-STATE.md)
  2. Freddy paste (if needed) in a code block
  3. "Ready. What are we working on?"
```

### The startup question

Marc asks once, concisely:

```
Session state: v1.20.74, master clean, 4 CRITICAL findings remaining.

Which roles this session?
  1. Full team — Freddy + Coach + Marc (Briefs, Plans, implementation)
  2. Code team — Coach + Marc (codebase work, no analyst)
  3. Quick fix — Marc solo (small changes, deploys)
  4. Custom — tell me which roles
```

Jon picks a number or types a custom answer. Marc proceeds accordingly.

### What Marc generates per role

**If Freddy is active:**
Marc combines FREDDY.md + SESSION-STATE.md into a single paste, displayed in a fenced code block:

```
Here's the Freddy onboarding paste. Copy this into your new Claude.ai session:

\`\`\`
[contents of FREDDY.md]

---

[contents of SESSION-STATE.md]
\`\`\`
```

**If Coach is active:**
Marc displays a one-liner:

```
Coach: read CLAUDE.md, SESSION-STATE.md, and COACH.md on startup.
No paste needed — you have repo access.
```

Jon copies this into Coach's terminal as the first message (or just says "startup").

**If Marc solo:**
No generation needed. Marc reads SESSION-STATE.md and begins work.

| Config | Roles | When Jon uses it | Marc's output |
|--------|-------|------------------|---------------|
| Full team | F + C + M | Major features, milestones, Briefs/Plans pipeline | Freddy paste + Coach note |

All configurations start the same way: Marc reads state, asks the question, generates the appropriate output.

## 6. Edge Cases

### Jon changes role configuration mid-session

Marc can regenerate pastes at any time. Jon says "add Freddy" or "I need a Coach paste now." Marc re-reads SESSION-STATE.md (or regenerates if state has changed), produces the paste.

No state reset needed — Marc just generates the appropriate output for the new role.

### A role gets compacted mid-session

**Coach or Marc (repo access):** The compaction summary carries forward. On resumption, Coach/Marc reads SESSION-STATE.md + the compaction summary. The files haven't changed — they just need to re-establish context.

**Freddy (no repo access):** Jon would need to re-paste FREDDY.md + SESSION-STATE.md. However, FREDDY.md is designed to also work as a reorientation document, not just a fresh-start document. The "Recently Active Work" section provides enough context to resume.

Recommendation: Add a note to FREDDY.md's header: "This document works for both fresh starts and mid-session reorientation after context loss."

### SESSION-STATE.md is out of date

The staleness check catches this. If SESSION-STATE.md's date is older than the latest commit, Marc regenerates before using it. Worst case: Marc generates it fresh at startup (~15 seconds).

### FREDDY.md is out of date

Coach maintains FREDDY.md. The static sections (role identity, conventions, workflow) rarely change. The "Recently Active Work" section is the most perishable — but SESSION-STATE.md covers current state, so FREDDY.md only needs periodic updates (once per milestone or when conventions change).

**Update triggers:**
- New milestone begins (update "Recently Active Work")
- Communication pattern changes (update "Communication Conventions")
- New role or workflow change (update "Working Pattern")

### No BC token / offline start

Startup doesn't require BC access. All generation is from local repo files (git log, TODO.md, etc.). BC-dependent state (sync status, planning line counts) is excluded from SESSION-STATE.md by design.

## 8. Implementation Plan for Marc

### Phase 1: SESSION-STATE.md generation (SMALL, ~30 min)

1. Write the generation logic as a procedure Marc follows (not a script — it's 7 repo reads + a file write, simple enough for inline execution).
2. Generate the first SESSION-STATE.md from current repo state.
3. Commit SESSION-STATE.md.

### Phase 2: CLAUDE.md updates (SMALL, ~15 min)

1. Add the "Multi-role startup" subsection to Session Startup Procedure.
2. Add step 6b to Session Shutdown Procedure.
3. Commit CLAUDE.md changes.

### Phase 3: FREDDY.md update (SMALL, ~5 min)

1. Add the reorientation note to the header.
2. Verify "Recently Active Work" section is current.
3. Commit if changed.

### Phase 4: Dry run (SMALL, ~10 min)

1. Marc simulates a startup: reads SESSION-STATE.md, asks the role question, generates a Freddy paste.
2. Jon pastes into a fresh Claude.ai session and confirms Freddy orients correctly.
3. Adjust paste format if needed.

**Total effort: ~1 hour. All phases are SMALL.**

No code changes to app.jsx or any source files. This is entirely documentation and procedure.

## 9. Open Questions for Jon

1. **Should the role question be asked every session, or only when Jon says "startup"?**
   Option A: Always ask on first message (could be annoying for quick solo fixes).
   Option B: Only when Jon explicitly says "startup" or "team session."
   Recommendation: Option B — Marc assumes solo mode unless Jon invokes startup.

2. **Should SESSION-STATE.md be committed to the repo or kept local?**
   Option A: Committed (visible in git, available to all sessions).
   Option B: Local-only (in .claude/ or similar, not tracked).
   Recommendation: Option A — Coach needs to read it, and it's useful context in git history.

3. **Should the Freddy paste include the full FREDDY.md or a trimmed version?**
   The current FREDDY.md is ~2K tokens. That's already concise. But if it grows, we might want a "paste version" vs "reference version."
   Recommendation: Paste the full file for now. Revisit if it exceeds ~4K tokens.

4. **Should Coach's startup also read SESSION-STATE.md automatically, or only on request?**
   Currently Coach reads CLAUDE.md + COACH.md on startup. Adding SESSION-STATE.md to the standard read list means Coach always has current state.
   Recommendation: Add to Coach's standard startup reads.

5. **Should the Close Out procedure auto-generate SESSION-STATE.md, or should it be manual?**
   Recommendation: Auto-generate as part of Close Out. Forgetting to update it is the main staleness risk. Making it automatic eliminates that.

6. **Naming: "startup" vs "/startup" vs something else?**
   A slash command (/startup) would be cleaner but requires skill registration. A plain word ("startup") works today with no tooling changes.
   Recommendation: Start with plain "startup" (v1). Add /startup skill later if the pattern sticks.

## 10. What This Does NOT Solve

- **Real-time coordination between sessions** — This system handles session start/end, not mid-session communication. Jon still copy-pastes between sessions manually. TODO #76 (multi-Claude coordination layer) addresses this separately.
- **Automatic role detection** — Marc can't know which role it is without Jon telling it. The system relies on CLAUDE.md being loaded into the correct environment.
- **Shared memory across roles** — Each Claude instance has its own conversation context. The repo files (SESSION-STATE.md, FREDDY.md, COACH.md) are the shared state mechanism. There is no real-time sync.
